Tour Configuration

The entire configuration for tours is stored in YAML files. Each TYPO3 package is scanned to check if it contains a .yaml file inside the Configuration/Tours/ folder. It can be one of more files, and they all will be processed. The file must follow the structure described in the next sections.

Configure tour

Each tour should be configured in a separate YAML file. The following properties are available:

identifier

identifier
Type
string
Required

true

Unique identifier for the tour. Use the format vendor/tourName.
The typo3 vendor is reserved for core tours.
It is recommended to use the name of the backend module (if applicable) as the tour name.
Examples: typo3/webLayout or vendor/webNewsManagement.

title

title
Type
string
Required

true

Title of the tour. It will be displayed in the Backend Module, where the user can see all available tours.

description

description
Type
string

Short description of the tour. It will be displayed in the Backend Module, where the user can see all available tours.

permission

permission
Type
array
Permissions required to access the tour.
The user must have all the listed permissions to see the tour.
The permissions are checked against the backend user and groups.
If the user does not have the required permissions, the tour will not be displayed.

Allowed permissions: 
modules: []
pagetypesSelect: []
tablesModify: []
tablesSelect: []
nonExcludeFields: []
explicitAllowdeny: []
filePermissions: []
availableWidgets: []
Copied!

Note

Not all of those permissions will make sens to be used, but we support them to be consistent with the TYPO3 backend.

moduleName

moduleName
Type
string
Module name to which the tour is bound. This is used to display the tour only when the user is in the corresponding module.
This should be a real module name as defined in the TYPO3 backend. Check the Configuration -> Backend modules module, to see the list of available modules.
Examples: web_layout, web_list, file etc.

Note

This module name is used to check if the user has access to the module. Read more about tour access in the chapter Tour Access.

isStandalone

isStandalone
Type
bool
Indicates whether the tour is standalone (i.e., a single step not bound to any element).
Such tours are rendered automatically when the user logs in to the backend (they should not be bound to any module).
Their purpose is to provide general information, welcome messages, or similar content.

skipButtonText

skipButtonText
Type
string
Custom text for the Skip step button.
Translations are supported. For example, LLL:EXT:typo3-interactive-tour/Resources/Private/Language/locallang.xlf:tour.skipButtonText.
If not set, the default text is used.

previousButtonText

previousButtonText
Type
string
Custom text for the Previous step button. Each step can override this setting.
Translations are supported. For example, LLL:EXT:typo3-interactive-tour/Resources/Private/Language/locallang.xlf:tour.previousButtonText.
If not set, the default text is used.

nextButtonText

nextButtonText
Type
string
Custom text for the Next step button. Each step can override this setting.
Translations are supported. For example, LLL:EXT:typo3-interactive-tour/Resources/Private/Language/locallang.xlf:tour.nextButtonText.
If not set, the default text is used.

finishButtonText

finishButtonText
Type
string
Custom text for the Finish button. Each step can override this setting.
Translations are supported. For example, LLL:EXT:typo3-interactive-tour/Resources/Private/Language/locallang.xlf:tour.finishButtonText.
If not set, the default text is used.

nextTourIdentifier

nextTourIdentifier
Type
string
Identifier of the next tour to be started after this one finishes.
This is useful to create a sequence of tours that guide the user through different parts of the backend.
The user must have access to the next tour for it to be started.

Note

Even is set, not always the next tour will be started. It depends on the user's permissions. Read more about tour access in the chapter Tour Access.

steps

steps
Type
array
Required

true

Array of steps for the tour. See Configure steps.

Example:

identifier: vendor/shopProducts
title: 'Introduction to Products Management Module'
description: 'This tour introduces the basics of the Products module. Learn how to manage your product listings.'
moduleName: 'shop_products'
isStandalone: false
enableInteraction: true
previousButtonText: 'Back'
nextButtonText: 'Next'
finishButtonText: 'Done'
nextTourIdentifier: vendor/shopCategories
steps:
  - title: 'Product List View'
    content: 'This section displays all the products in your catalog. You can filter, sort, and search through the list.'
    target: '.products-list-container'
    frame: 'content'
    side: 'top'
    align: 'start'
    enableInteraction: false
  - title: 'Add New Product'
    content: 'Click this button to create a new product entry.'
    target: 'button.new-product-button'
    frame: 'content'
    side: 'right'
    align: 'center'
    enableInteraction: true
  - title: 'Product Details'
    content: 'Here you can enter details like product name, description, and price.'
    target: '.product-form'
    frame: 'content'
    side: 'bottom'
    align: 'center'
    enableInteraction: false

Configure steps events

Each step can contain an events property to define custom JavaScript events. These events can be assigned to 3 different groups:

  • onShow: events that are triggered when the step is shown
  • onExit: events that are triggered when the step is hidden
  • userActions: events that should be triggered by the user

Note

The userActions events will not be required if the requireUserActions property is set to false on step level
